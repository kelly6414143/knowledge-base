# Cursor Prompt 模板（前端工程師版）

> 記得在不確定的prompt情境下，最後一定要加上「有疑問請先提問在執行」

## 1. 小型重構（不改行為）【Ctrl + K】

```
請在「不改變任何功能與 UI 行為」的前提下，
重構這個 component，目標是：

1. 提升可讀性與結構清楚度
2. 抽離明顯重複的邏輯（如有）
3. 保留既有命名風格與專案寫法
4. 不新增任何新功能

只提供重構後的程式碼。
```

✅ 使用時機：你今天剛做的那種
❌ 不適合：核心業務邏輯

---

## 2. 重構後安全檢查（Review）【Ctrl + L】

```
請檢查這次重構是否可能改變行為，
特別注意以下幾點：

- useEffect 的依賴與觸發時機
- 事件處理與條件判斷
- render 分支是否與原本一致

若有風險，請指出具體行數與原因；
若無，請列出你確認過的重點。
```

✅ 當提交前保險用

---

## 3. TypeScript 錯誤解析（快速定位）【Ctrl + L】

```
請解釋這個 TypeScript 錯誤的真正原因，
並指出是哪個型別或欄位不一致造成的。
請給出最小修改的修正方式。
```

✅ 命中率極高
✅ 比自己看 TS error 快很多

---

## 4. API 型別變更 → 跨檔同步【Ctrl + K】

（先多選相關檔案）

```
這個 API response 結構已調整，
請同步更新相關的 type、hook 與 component 使用處。

請保持既有資料流與命名方式，
不要改變原本的行為。
```

👉 這是 Cursor Pro 的殺手級用法

---

## 5. Code Review（找未來風險）【Ctrl + L】

```
請以資深前端工程師角度 review 這段 code，
指出可能的維護性問題、隱性風險，
或未來擴充時可能遇到的困難。
```

適合：

- effect 很多
- state flow 複雜
- 老 code

---

## 6. 效能檢查（不做過度優化）【Ctrl + L】

```
請檢查這段程式碼是否有明顯的效能風險，
例如不必要的 re-render、effect 重複執行等。
請只指出「值得關注」的地方，不要做過度優化。
```

---

## 7. 只解釋，不改 code【Ctrl + L】

```
請逐段解釋這段程式碼在做什麼，
並說明資料流與主要狀態變化。
請不要修改任何程式碼。
```

適合：

- 接手別人寫的 code
- 看老專案

---

## 8. 讀專案【Ctrl + L】

```
請幫我說明這個專案的整體架構：

1. 狀態管理方式

2. API 呼叫集中在哪

3. 共用元件的設計原則
```

---

## 8. 防呆【Ctrl + L】

```
你是一位熟悉本專案架構的前端工程師，
請在不改動 API contract 的前提下，
優化這段 React 邏輯的可讀性與可測試性
```

## 9. 刪除用不到的檔案

```
找出沒有用到的檔案, 先提供給我, 等我確認OK後, 在進行刪除
```

## 10. 優化專案架構

step1:現有架構

```
請掃描整個專案，將專案的架構呈現出來
```

> 目標架構分為
>
> > A. Feature-based（推薦：大專案、多人協作）  
> > 以「功能」為單位切資料夾：features/<feature>/components|hooks|api|store|types  
> > 優點：同功能的程式碼聚在一起，改一個功能不會全專案找檔案  
> > 很適合你這種有「存款/優惠稽核、VIP、活動」等模組的產品
>
> > B. Layer-based（適合小專案或 UI library）  
> > 以「層」切：components/ hooks/ pages/ services/ store/ utils/ types  
> > 優點：初期直覺  
> > 缺點：越大越容易變「超大資料夾」，後期不好管

step2:架構診斷報告

```
你是資深前端架構師。請根據我目前專案的目錄結構，找出：
1) 重複/命名不一致的資料夾
2) 職責混雜（例如 api/logic/ui 混在一起）的區域
3) 過深/過淺的目錄層級問題
4) 建議的目標架構（以 feature-based /layer-based 為主），並提供搬移順序（分批做）
輸出請包含：現況問題清單、目標 tree、以及每批要改哪些資料夾。
```

step3:目標 tree + 命名規則

> Cursor明確給規則，例如：  
> 資料夾命名：kebab-case 或 camelCase（擇一）  
> 檔名規則：React component PascalCase.tsx / Vue PascalCase.vue  
> 單一職責：UI / domain / api / store 分開  
> import 邊界：feature 內可互相引用，但跨 feature 只能走 shared/ 或 core/

```
請給我一份「專案檔案結構規範」：
- 目標資料夾 tree（含 shared/core/features）
- 命名規則（資料夾/檔案/元件）
- import 邊界（哪些能互相依賴，哪些不行）
- 以及我們要用 ESLint 或路徑別名來落實的建議
```

```
}
```
